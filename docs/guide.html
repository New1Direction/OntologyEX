When to use it

Use it when fuzzy knowledge is costing you speed.

The skill is most useful when a system needs shared meaning: AI tools, search, data models, documentation, workflows, or market research.

AI agents

How it works

It keeps four layers connected.

The power is not the layers by themselves. The power is the mapping: every app object maps to a domain concept, every domain concept anchors to a universal category, and every task consumes or produces domain concepts.

L0

Upper layer

Basic categories that work in any business: Agent, Event, Process, Object, Role, Place, Time, Quantity, Information Artifact.

L1

Domain layer

The nouns of the field. In payments: Charge, Refund, Dispute. In publishing: Article, Comment, Tag.

Test

L2

Task layer

The verbs and workflows: process a refund, publish an article, follow a member, resolve a market.

Test

L3

Application layer

The actual system artifacts: DB tables, API objects, tool schemas, events, UI objects, fields, routes.

Workflow

The skill works middle-out.

It does not start with philosophy and it does not drown in app details. It mines the domain first, then attaches tasks and system objects.

Scope

Name the target, the consumer, and what is in or out of bounds.

Ask

Write competency questions: what must the finished map be able to answer?

Mine

Pull terms from docs, APIs, schemas, routes, events, tests, job posts, and standards.

Anchor

Select upper categories from gist, schema.org, BFO/CCO, or another known ontology.

Model

Define domain classes, relations, task verbs, preconditions, and effects.

Bind

Map app tables, API types, events, UI objects, and tools back to domain concepts.

Validate

Run the script to catch duplicates, dangling references, missing anchors, and orphan classes.

Emit

Generate the binding your consumer needs: MCP tools, JSON-LD, RDF, code types, diagrams, or RAG metadata.

How it gives you an edge

It turns vague expertise into reusable leverage.

The advantage is not "having an ontology" as a trophy. The advantage is that your agents, search, schemas, and people all speak the same controlled language.

1. Agent precision

Fewer bad tool calls

Tasks include preconditions and effects. That gives an AI agent guardrails before it mutates data or calls an external API.

2. Faster due diligence

See a business clearly

Mining docs, APIs, help centers, pricing pages, and job posts reveals how a company really thinks about its domain.

3. Better retrieval

Cleaner RAG filters

Domain classes and task labels become metadata. That means sharper search, fewer irrelevant chunks, and better answer grounding.

4. Schema intelligence

Find gaps and dead weight

When an app table or field cannot map to a domain concept, it is either app-only, missing from the model, or a cleanup candidate.

5. Interop

Borrow standards

Reusing schema.org, gist, FIBO, FHIR, GS1, PROV-O, or other standards gives you compatibility instead of a private vocabulary nobody else understands.

6. Team alignment

Less talking past each other

Product, engineering, data, support, and AI agents get the same definitions, relations, and workflow names.

Quality checklist

Good output has evidence and mappings.

If the ontology cannot answer its own competency questions, or if the app layer is not mapped upward, it is not done yet.

  • Every domain class has a definition, source, and upper anchor.
  • Every relation has domain, range, cardinality, and evidence.
  • Every task has actors, inputs, outputs, preconditions, effects, and source.
  • Every app concept binds to an L1 domain class.
  • Synonyms are collapsed into one canonical id.
  • Published standards are reused when they fit.
  • The validator passes or the warnings are understood.
  • The final output matches the consumer: MCP, RAG, graph, DB/API, or docs.
